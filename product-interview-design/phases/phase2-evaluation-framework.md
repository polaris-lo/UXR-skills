# Phase 2：评估框架

你正在执行产品研究方案设计的 Phase 2。你的目标是定义将要测量什么——在设计任务或访谈问题之前。

---

## 输入

开始之前，先阅读 **Phase 1 研究框架文档**。每个框架维度必须能追溯到 Phase 1 中至少一个研究问题。

---

## 需要从用户处获取的信息

询问用户：

1. **哪些体验维度最关键？** 基于研究问题，最需要评估产品的哪些方面？
2. **是否已有维度列表或指标体系？**（如果用户已有现成的评估指标，从这里出发进行审查，而不是从零构建。）
3. **研究是否包含评分环节？** 参与者会对维度打分，还是纯定性研究？
4. **这是多产品对比研究吗？** 如果是，将对比哪些产品或版本？

---

## 方法论指导

**核心区分**：评估框架不等于访谈提纲。框架规定*测量什么*（供研究者使用）；提纲规定*如何收集证据*（问题的措辞）。两者必须分开——如果框架维度的措辞出现在访谈问题中，参与者会给出"对照评分标准的答案"而不是体验叙述。

**核心张力**：维度必须具体到能产生可操作发现，同时又要足够抽象，使参与者不会察觉正在被测量什么。一旦用户猜到你在测量什么，他们就会进入"评测模式"而不是叙述体验。

**框架结构——三个层级：**

*第一层：主要领域* — 3–5 个与业务问题对应的核心领域。对于每个领域：如果它产出了强有力的发现，接下来会做出什么产品决策？如果没有发现，会损失什么？删除任何即使有发现也不会推动任何行动的领域。

*第二层：维度* — 每个领域 2–5 个维度。每个维度必须具备：
- 1–2 句分析性语言的定义（供研究者使用，不面向用户）
- 可观察证据：用户说什么或做什么，能说明该维度有效或失效？
- 收集优先级：P0（核心）、P1（重要但次要）、P2（情境性/观察性）

*第三层：每个维度的收集方式*

| 优先级 | 收集方式 |
|--------|--------|
| P0 | 评分 + 评分后追问（定量信号 + 定性解释） |
| P1 | 评分 + 低分时简短跟进 |
| P2 | 任务过程中观察，或会话结束时简短提问 |

**关于评分量表（如适用）**：使用描述性锚点量表——定义每个评分点（1、3、5）对应的可观察行为。抽象锚点（"1 = 非常不满意"）会导致参与者之间的理解不一致。要求参与者在打分时出声思考。

**关于多产品对比**：框架必须在所有产品间保持一致。提前明确：不适用的维度是跳过、标记为 N/A，还是排除在对比之外。

---

## 你的任务

根据用户提供的信息，起草一份**评估框架文档**，包含以下几节：

1. **领域地图** — 3–5 个领域及每个领域一行的设置理由
2. **维度表** — 每个维度：名称、定义、可观察证据、收集方式（P0/P1/P2）
3. **评分量表**（如适用）— 每个 P0 维度的描述性锚点
4. **多产品说明**（如适用）— 维度适用性或对比限制

标注任何无法清晰追溯到 Phase 1 研究问题的维度——这些是候选删除项。

---

## 完成本 phase 后

将完整的评估框架文档呈现给用户，然后询问：

- 每个 P0 维度是否都对应了 Phase 1 中的某个研究问题？
- 是否有即使产出发现也不会改变任何产品决策的维度？（如果有，删除。）
- 有没有维度的措辞听起来像是可以直接问参与者的问题？（如果有，需要重写——那些措辞应该出现在研究者的备注里，而不是访谈中。）

**在用户明确 approve 本文档之前，不要进入 Phase 3。**
