# product-interview-analyst（产品访谈分析师）

针对产品评测访谈数据的系统性定性分析方法。引导你完成沉浸式阅读、指标对齐编码、模式解读和可落地报告生成的全流程。专为商业业务场景设计：快速周转、多受访者频率追踪，产出直接支持产品决策的分析结果。

---

## 核心理念

**目标不是提取用户说了什么——而是理解这意味着什么，以及应该怎么做。**

四条原则贯穿整个工作流：

1. **先认识人，再编码数据。** 谁说了某句话，决定了这个发现究竟是普遍性产品问题、场景限定的边缘案例，还是个人特殊情况。这个判断离开 Phase 1 无法完成。

2. **代码必须解释机制，而不仅仅是贴标签。** "导航不清楚"是标签。"当前页面在层级结构中缺乏位置标识，导致用户在多层级浏览后无法定位返回路径，被迫从首页重新开始"才是机制。只有机制性代码才能产出可落地的产品指导。

3. **用户说的话与你的解读必须分开。** 用户原话是证据，分析结论是你的推断，两者不能混同。始终展示证据，始终标注解读从哪里开始。

4. **矛盾是数据，不是噪音。** 当两位受访者对同一功能给出截然相反的评价时，不要平均化——要追问为什么。答案几乎总是比任何单一数据点更有价值。

---

## 这个 Skill 不是什么

本 Skill 借鉴定性研究的方法论原则，但专为**商业分析**而设计，而非学术研究。具体来说：

- 不涉及理论框架综述（无学术理论，无文献综述）
- 不使用学术质量指标（无科德框架，无评分者信度）
- 不产出学术写作格式
- 输出物是产品报告和可行动摘要，而非论文

开始前请先建好项目文件夹：

```
your-project/
├── interviews/          # 访谈文本（每位受访者一个文件）
├── analysis/
│   ├── phase1-reports/
│   ├── phase2-reports/
│   ├── phase3-reports/
│   ├── phase4-reports/
│   └── codes/
└── memos/               # 可选：你自己的分析笔记
```

## 工作流概览

| 阶段 | 名称 | 目的 | 产出 |
|------|------|------|------|
| 1 | 快速沉浸 | 编码前建立每位受访者的认知框架 | 沉浸备忘录、跨用户信号 |
| 2 | 指标对齐编码 | 系统性编码，对齐评估维度 | 代码本、频率矩阵、矛盾日志 |
| 3 | 模式解读 | 从"是什么"跃升到"为什么"和"那又怎样" | 优先级矩阵、机制命题、矛盾分析 |
| 4 | 产品报告 | 产品/设计团队可用的输出 | 结构化报告、引语证据表 |

每个阶段结束后暂停，等待你确认后再继续。

---

## 本 Skill 的执行方式

本 Skill **逐 phase 执行**。每个 phase 结束时：

1. 完整呈现该 phase 的交付物
2. 总结已做出的关键决策和任何待解决的问题
3. 明确询问用户是否 approve 再继续
4. **在用户确认之前，不要开始下一个 Phase**

这很重要，因为分析决策是累积的。Phase 1 中一份不够详细的沉浸备忘录，会让 Phase 2 的代码缺乏情境，进而让 Phase 3 的命题归因出错。逐 phase 发现错误只需几分钟，在完整分析完成后才发现则意味着整个流程需要重来。

---

## 模型建议

- **Phase 1**：Sonnet——仔细阅读，撰写备忘录
- **Phase 2**：Sonnet——系统处理，构建代码本
- **Phase 3**：Sonnet 或 Opus——模式推理，矛盾分析（如命题深度不够可换用 Opus）
- **Phase 4**：Sonnet——基于 Phase 3 产出生成报告

## 启动步骤

开始前请确认：

1. 访谈文本是否已放入 `interviews/` 目录？（建议每位受访者一个文件）
2. **是否有定量评分数据**（如受访者对各产品维度进行了 1–5 分评分）？如有，在 Phase 1 开始前导入（见下方"定量背景导入"）。
3. 是否有预设的评估框架（如产品维度清单或评分标准），或者需要从数据中归纳编码框架？
4. 本次分析要回答什么研究问题或支持什么业务决策？

确认后进入 Phase 1。

## 可选前置步骤：定量背景导入

**适用条件**：受访者在访谈之外还填写了结构化评分工具（李克特量表、维度评分等）。

**操作方式**：将每位受访者的评分附加到其 Phase 1 沉浸备忘录中。这不是单独的定量分析——而是在后续两个阶段中强化定性解读的上下文：

- **在 Phase 2 中**：当某受访者对某维度打了低分时，优先寻找解释该低分的定性代码
- **在 Phase 4 中**：将维度评分与定性代码结合，产出对业务干系人最具说服力的证据格式：
  > *"6 位受访者中有 4 位提到了这个问题；这 4 人对该维度的平均评分为 2.1/5，而其余 2 人为 4.3/5。"*

这种格式同时回答了干系人必问的两个问题：**"有多少人有这个问题？"**和**"他们认为有多严重？"**——一个数字，可追溯到访谈文本和评分数据两个来源。

## 核心概念

### 个别现象 vs. 共性现象

频率高低本身不决定重要性。只有一位受访者提到的发现，可能是揭示根本设计缺陷的高影响边缘案例——前提是该受访者的情境代表许多用户的真实场景。反之，所有受访者都提到的发现，可能只是一个没有可行解决方案的表面抱怨。

始终将频率与 Phase 1 中的受访者背景**结合**解读。这正是 Phase 1 必须先于 Phase 2 的原因——不知道是谁说的，就无法判断这意味着什么。

### 陈述（Accounts）vs. 解读（Interpretations）

**陈述**是受访者说的话：「找到我想要的东西要绕好几步。」
**解读**是你的推断：「多层级导航结构增加了认知路径长度，使有明确目标的用户仍需经历多次跳转才能到达目的地。」

两者都有价值，互不替代。所有分析产出都必须明确区分这两者。

### 机制性代码

弱代码：「受访者觉得搜索结果不相关。」

强代码：「筛选条件在跨页面操作后被系统重置，导致有明确目标的用户每次返回都需要重新设置条件——进而放弃使用筛选功能，改为手动逐条浏览。」

检验标准：产品团队能否基于这条代码做出具体的设计决策？能则达标。否则继续细化。

### 三层证据链

最终报告中的每条洞察都必须具备全部三层。缺少任何一层，业务干系人都有充分理由驳回这条发现：

| 层级 | 内容 | 防御对象 |
|------|------|---------|
| **数字层** | 覆盖率（N/总数）+ 显著度比例 + 评分对比（如有） | "就这一个人这么说吗？"/"有多严重？" |
| **原话层** | ≥ 2 条逐字引语，附受访者背景说明 | "你是不是过度解读了？"/"给我看证据" |
| **机制层** | 为什么会发生，是怎么运作的 | "那又怎样？我们应该做什么？" |

只有数字没有故事。只有引语是个例。只有解读没有证明。三层齐备，才是难以被驳回的发现。
